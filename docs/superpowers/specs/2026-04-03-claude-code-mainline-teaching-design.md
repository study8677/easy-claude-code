# Claude Code 源码教学主线化改造设计

## 背景

当前仓库已经具备较强的资料密度：

- `examples/` 提供 9 个可运行示例
- `docs/paths/` 提供阶段式学习路径
- `docs/layers/` 提供分层深挖
- `docs/source-map*` 与 `docs/example-source-bridge*` 提供源码导航

但这些材料的默认阅读视角仍偏向“分层”“分文档”“分主题”。

对初学者而言，最大的阻力不是资料不够，而是**抓不住 Claude Code 的运行主线**。读者容易出现以下问题：

1. 知道仓库里有很多材料，但不知道先抓哪条链。
2. 跑完 example 之后，仍然不知道它在真实源码里处于什么位置。
3. 能记住局部概念（如 tools、permissions、streaming），但无法把它们串成一次真实请求的生命周期。
4. 打开源码文件后，不知道当前这一步的输入、输出、边界和下一跳。

因此，这次改造的目标不是继续扩写资料，而是**把现有资料统一到一条强主线教学模型上**。

---

## 目标

把整个仓库重构为：

- **1 条绝对主线**
- **2 条辅助理解线**

并让 README、paths、source-map、layers、examples 全部围绕这套模型协同工作。

最终，读者应能稳定做到：

1. 复述 Claude Code 中“一次用户请求如何流动”。
2. 知道自己当前在主线的哪一步。
3. 知道下一步该打开哪个源码文件、搜索哪个 symbol。

---

## 非目标

本轮设计不追求：

- 新增大量全新主题或额外高级专题
- 把仓库改造成 Claude Code 的逐行复刻
- 一次性重写全部文档内容
- 增加复杂交互式站点、可视化前端或新运行时依赖

本轮只聚焦一件事：**让教学结构更强地主线化**。

---

## 核心教学模型

### 绝对主线：单次请求生命周期

仓库今后的第一教学视角统一为：

**用户输入  
→ 启动 / 模式分流  
→ 进入 `query` / `queryLoop`  
→ 模型产出  
→ 工具调用  
→ 工具执行  
→ 工具结果回流  
→ 状态 / UI 更新  
→ 下一轮或退出**

这条链是读者理解 Claude Code 的“骨架”。

仓库中的所有核心材料，都应优先回答：

- 我现在正在主线的哪一步？
- 这一步在源码里发生在哪？
- 它的输入是什么？
- 它的输出是什么？
- 它下一步流向哪里？

### 辅助线 1：源码导航线

在主线每一步上，补充读者最需要的源码落点信息：

- 先开哪个文件
- 再搜哪个 symbol
- 为什么先看这里而不是别处

辅助线 1 的任务是：**把教学主线变成实际可追踪的源码路径**。

### 辅助线 2：架构理解线

在读者已经知道“发生了什么”之后，再解释：

- 为什么这一层存在
- 它解决什么工程问题
- 与 permissions / streaming / memory / UI 的关系是什么

辅助线 2 的任务是：**把运行主线升级成工程理解**。

## 信息架构重定位

### 1. README / README_EN

#### 新职责

README 不再主要按“仓库里有什么资料”组织，而是先承担：

1. 定义唯一主线
2. 展示主线总图
3. 说明 9 个 examples 各自位于主线的哪个节点
4. 给出第一轮阅读的最短路径

#### 期望效果

读者进入仓库首页后，应立即理解：

- Claude Code 学习的第一问题不是“有哪些层”，而是“请求怎么流动”
- 当前仓库已经为这条主线准备了示例、路径、导航与深挖材料

### 2. docs/paths/

#### 新职责

`docs/paths/` 作为“课程主线”，应从“阶段式阅读路线”进一步升级为：

- **按主线分段推进**
- 每阶段只覆盖主线中的一小段
- 明确告诉读者：这一阶段先不要看什么

#### 期望效果

读者不再被鼓励同时打开很多材料，而是：

- 在本阶段只搞懂 2~3 个因果关系
- 只沿主线向前推进一小步

### 3. docs/source-map*

#### 新职责

`source-map` 变成“主线到源码的追踪图”，而不是纯资料总览。

每个主线步骤都应提供：

- 对应文件
- 推荐搜索的 symbol
- 当前步骤的输入
- 当前步骤的输出
- 下一步流向

#### 期望效果

读者卡在某一步时，能立刻用 `source-map` 找到源码落点，而不是继续靠猜。

### 4. docs/example-source-bridge*

#### 新职责

桥接页进一步承担“从 example 回到主线，再从主线回到源码”的任务。

不只告诉读者“跑完这个去看哪篇文档”，还要明确：

- 这个 example 在主线的哪一段
- 它省略了哪些真实复杂度
- 源码里应先开哪些文件

### 5. docs/layers/*

#### 新职责

每篇 layer 文档不应再像独立专题，而应成为主线的“局部解释器”。

每篇文档优先回答：

- 这一层在主线里的职责是什么
- 如果没有这一层，主线会断在哪
- 这一层最容易被误解成什么

### 6. examples/*

#### 新职责

每个 example 不再只是“第 N 层示例”，而要明确成为**主线切片演示**。

例如：

- `l1`：入口与模式分流
- `l2`：单轮 loop 核心
- `l3`：工具调用如何接入主线
- `l7`：权限在主线哪个关口介入
- `l8`：streaming 如何贯穿主线
- `l9`：context 管理如何影响主线持续运行

## 统一教学模板

为减少读者上下文切换成本，关键文档应统一增加固定栏目。

### README 固定栏目

1. 你现在要抓的唯一主线
2. 主线总图
3. 9 个 examples 在主线中的位置
4. 第一次阅读只看什么

### Path 页面固定栏目

1. 本阶段只搞懂什么
2. 你在主线中的位置
3. 先跑哪个 example
4. 再读哪篇 layer
5. 再开哪些源码文件
6. 这一阶段先不要看什么

### Source Map 固定栏目

1. 主线步骤
2. 对应文件
3. 建议先搜的 symbol
4. 这一步的输入
5. 这一步的输出
6. 下一步流向

### Layer 文档固定栏目

1. 这一层在主线里的职责
2. 如果没有这一层，主线会断在哪
3. 读源码时只需要盯哪几个 symbol
4. 这一层最容易误解什么
5. 读完后你应该能回答什么问题

### Example docstring 固定栏目

1. 这个 example 在主线里的位置
2. 它模拟了哪一步
3. 真实源码在哪
4. 跑完后下一步该去哪里
5. 此时先不用管什么

## 实施顺序

### Phase 1：先把主线立起来

优先修改：

- `README.MD`
- `README_EN.MD`
- `docs/paths/README.md`
- `docs/paths/README.en.md`

#### 目标

先让读者进入仓库时就不迷路，建立“唯一主线”的第一印象。

### Phase 2：把主线落到源码导航

接着修改：

- `docs/source-map.md`
- `docs/source-map.en.md`
- `docs/example-source-bridge.md`
- `docs/example-source-bridge.en.md`

#### 目标

让主线具有实际可追踪的文件和 symbol 落点。

### Phase 3：把课程页改成按主线分段推进

然后修改：

- `docs/paths/p1-first-hour*.md`
- `docs/paths/p2-core-loop*.md`
- `docs/paths/p3-source-reading*.md`
- `docs/paths/p4-advanced-architecture*.md`

#### 目标

让课程页真正承担“教学节奏控制器”的角色。

### Phase 4：统一 layers 与 examples 的主线模板

最后修改：

- `docs/layers/*.md`
- `docs/layers/*.en.md`
- `examples/*.py` 顶部 docstring

#### 目标

让整个仓库在细节层面维持相同的教学语言与提示结构。

## 成功标准

本轮改造完成后，应该能通过以下标准判断是否有效：

### 读者层面

1. 能用自己的话描述“一次请求如何穿过 Claude Code”
2. 看到任一 example / path / layer 时，能判断它在主线中的位置
3. 打开源码时，不再只是“知道文件名”，而是知道为什么此时该看这个文件

### 仓库表达层面

1. 首页不再是资料堆介绍，而是教学主线入口
2. paths 不再只是阶段目录，而是主线推进器
3. source-map 不再只是静态导图，而是调用链追踪工具
4. layers 与 examples 使用统一的主线语言

---

## 风险与约束

### 风险 1：过度重写导致信息损失

为降低风险，应坚持“重构表达，不重构事实”：

- 不改变源码事实
- 不改变 examples 的教学定位
- 只改变资料的组织方式与默认视角

### 风险 2：主线过强，横向主题被压扁

解决方式：

- 维持主线优先
- 但保留 permissions / streaming / memory / UI 等专题深挖作为辅助线材料

### 风险 3：中英双语版本失衡

解决方式：

- 设计上先确定结构模板
- 中英文件按同一信息架构同步调整

---

## 设计结论

这次改造的本质，不是继续增加“更多教学内容”，而是把仓库变成一个更强的**源码主线教学系统**。

教学顺序将从：

**看到很多资料 → 自己拼主线**

改造成：

**先抓住单次请求生命周期 → 再落到源码文件与 symbol → 再理解架构边界**

如果这套设计被正确落地，仓库会更像一门真正的 Claude Code 源码课程，而不是一组优秀但相对分散的学习材料。
